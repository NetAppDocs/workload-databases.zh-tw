= 在 Workload Factory 中實作架構良好的資料庫配置
:allow-uri-read: 
:icons: font
:imagesdir: ./media/


[role="lead"]
利用配置分析洞察和建議，借助 NetApp Workload Factory，您可以針對 Microsoft SQL Server 和 Oracle 資料庫配置實作最佳實務。您可以輕鬆查看架構良好的狀態，了解資料庫配置中存在的問題，並修復那些在可靠性、安全性、效率、效能和成本方面未達到最佳狀態的配置。

您也可以忽略不適用於您環境的組態分析，以避免產生額外的警示和不準確的結果。

link:optimize-overview.html["了解 Workload Factory 中的配置分析和良好架構狀態。"]



== 關於這項工作

Workload Factory 每日分析資料庫組態。每日分析提供架構完善的狀態、洞察分析與建議。您可以自動修復組態問題，以符合最佳實務做法。

您可以在 Workload Factory 控制台的「資料庫清單」或「架構完善」儀表板中檢視和修復問題。

您也可以選擇執行一次性評估，以審查資料庫組態的最佳實務做法，而無需儲存認證資料或註冊資源。

.分析內容
Workload Factory 分析以下配置的良好架構狀態：

對於 Microsoft SQL Server 執行個體：

* 儲存規模：包括儲存層、檔案系統保留空間、記錄磁碟機大小及 TempDB 磁碟機大小
* 儲存配置：包括使用者資料檔案放置、記錄檔放置及 TempDB 放置
* 儲存配置：包含容量管理、精簡配置、分層策略、快照、Microsoft 多路徑 I/O (MPIO) 狀態和 MPIO 逾時設置
* 計算：包括調整大小、作業系統修補程式和網路適配器設置，如接收方縮放 (RSS)、TCP 卸載和 MTU 對齊
* 應用程式：包括 Microsoft SQL Server 授權、Microsoft SQL Server 修補程式和 MAXDOP 設定
* 彈性：包含本機快照、FSx for ONTAP備份、跨區域複製 (CRR) 和 Microsoft SQL 高可用性。
* 克隆：包括刷新和刪除在 Workload Factory 內部或外部創建的超過 60 天的克隆（沙盒）的選項


對於 Oracle 資料庫：

* 儲存空間大小：包括交換空間分配和檔案系統餘裕
* 儲存配置：包含容量管理、精簡配置、分層策略、快照、儲存效率以及使用 NFS 或 iSCSI（無論是否啟用自動儲存管理（ASM））部署的作業系統配置，包括 Microsoft Multipath I/O（MPIO）狀態和設定，以及下列 dNFS 設定：dNFS 啟用、dNFS 一致性 IP 解析、dNFS 設定檔和 dNFS `nosharecache`
* 儲存佈局：包括重做日誌放置、臨時表空間放置、資料檔案放置、存檔日誌放置、控製檔案放置和二進位檔案放置、ASM 磁碟組 LUN 數量
* 運算：包含作業系統修補程式
* 應用程式：包括 Oracle Critical Patch Updates
* 恢復能力：包括跨區域複寫（CRR）和應用程式一致的快照


Oracle 的架構完善分析功能會在組態依賴其他組態時通知您，並列出最佳化連結組態的動作。



== 對您的組態進行一次性評估

執行一次性評估，以了解您的 Microsoft SQL Server 執行個體或 Oracle 資料庫的架構是否良好，而無需儲存認證或註冊資源。

您可以多次執行指令碼並上傳結果。Workload Factory 會在每次上傳時更新架構完善的狀態和洞見。

[role="tabbed-block"]
====
.Microsoft SQL Server 執行個體
--
.步驟
. 使用其中一項登link:https://docs.netapp.com/us-en/workload-setup-admin/console-experiences.html["主控台體驗"^]入。
. 選擇選單 image:workloads-menu-icon.png["漢堡選單圖示用於導航至儲存、EDA、AI、資料庫、VMware 和管理等工作負載。"] 然後選擇 *Databases* 。
. 在 *Inventory* 中，選擇 *SQL Server* 作為引擎類型。
. 選擇 *One-time assessment* > *Learn about assessment*。檢閱有關一次性評估的資訊。
. 選擇 *Download script* 。
+
.. 將指令碼複製到您的 Microsoft SQL Server 主機。
.. 在 PowerShell 中運行腳本。一個腳本會收集所有實例的元資料。


. 選擇 *One-time assessment* > *Upload* 。找到檔案並將其上傳到 Workload Factory 主控台。
. 檢視任何組態的建議。Workload Factory 的建議中包含最佳實務做法，並指出如果組態未最佳化可能出現的問題。


--
.Oracle 資料庫
--
.步驟
. 使用其中一項登link:https://docs.netapp.com/us-en/workload-setup-admin/console-experiences.html["主控台體驗"^]入。
. 選擇選單 image:workloads-menu-icon.png["漢堡選單圖示用於導航至儲存、EDA、AI、資料庫、VMware 和管理等工作負載。"] 然後選擇 *Databases* 。
. 在 *Inventory* 中，選擇 *Oracle* 作為引擎類型。
. 選擇 *One-time assessment* > *Learn about assessment*。檢閱有關一次性評估的資訊。
. 選擇 *Download script* 。
+
.. 將 Python 指令碼複製到您的 Oracle 資料庫主機。
.. 在對應的 shell 中執行腳本。一個腳本會收集所有資料庫的元資料。


. 選擇 *One-time assessment* > *Upload* 。找到檔案並將其上傳到 Workload Factory 主控台。
. 檢視任何組態的建議。Workload Factory 的建議中包含最佳實務做法，並指出如果組態未最佳化可能出現的問題。


--

== 在開始修復、關閉和重新啟動配置之前

持續監控資料庫環境並自動修正組態問題需要符合下列需求。一次性評估不需要這些需求。

* 你必須link:https://docs.netapp.com/us-en/workload-setup-admin/add-credentials.html["授予 _操作和修復_ 權限"^]在您的 AWS 帳戶中。
* 此資源必須在 Workload Factory 中註冊，且儲存類型必須為 FSx for ONTAP。link:register-instance.html["了解如何註冊資源。"]
* 在修改設定或組態之前，請檢閱每項建議。對於 RSS 和 MAXDOP，請在變更正式作業環境之前測試建議的設定以檢查效能。




== 修復配置問題

修正在 FSx for ONTAP儲存上執行的 SQL Server 或 Oracle 環境的設定問題。


NOTE: 修正組態可能會導致停機或服務中斷。在選擇修正組態之前，請務必仔細檢閱建議。

.步驟
. 使用其中一項登link:https://docs.netapp.com/us-en/workload-setup-admin/console-experiences.html["主控台體驗"^]入。
. 選擇選單 image:workloads-menu-icon.png["漢堡選單圖示用於導航至儲存、EDA、AI、資料庫、VMware 和管理等工作負載。"] 然後選擇 *Databases* 。
. 在*庫存*中，選擇引擎類型：Microsoft SQL Server 或 Oracle。
. 選擇要修復特定配置的資源。
+
** 如果選擇了 Microsoft SQL Server，請選擇「*實例*」標籤。
** 如果您選擇了 Oracle，請選擇「*資料庫*」標籤。


. 選擇“查看並修復”以查看實例的配置問題。
. 在實例的「良好架構狀態」頁面上，查看分析結果。
+
您可以按類別、子類別、狀態、嚴重性和標籤過濾配置問題。

+
選擇 *Export PDF* 以下載調查結果報告。

. 選取下拉式箭頭以檢視任何組態的建議。建議包括最佳實踐、未優化配置的潛在缺陷以及重要注意事項。請務必仔細檢閱建議內容。
. 當選項可用時，選擇“查看並修復”配置問題。
+
*全選*是預設設置，但您可以選擇特定資源進行修復。

+
.. 除了複製清理之外，對於所有組態，請檢閱建議詳細資料以瞭解修正問題的影響。某些補救作業可能會導致停機或服務中斷。
.. 若要清除複本，請選取要重新整理或刪除的複製資料庫（沙箱）。
+
*** 刷新克隆會使其與來源資料庫同步。刷新僅適用於在 Workload Factory 中建立的克隆。
*** 刪除克隆會將其永久刪除，釋放儲存空間並降低成本。您可以刪除在 Workload Factory 內部和外部建立的克隆。




. 選取 * 繼續 * 以修正組態問題。


.結果
Workload Factory 開始修復問題。選擇「作業監控」標籤來查看操作的狀態。



== 關閉資料庫組態分析

為避免不必要的警報和不準確的最佳化結果，請忽略不適用於您資料庫環境的特定資料庫配置分析。您可以隨時重新啟動已忽略的組態分析。

.步驟
. 使用其中一項登link:https://docs.netapp.com/us-en/workload-setup-admin/console-experiences.html["主控台體驗"^]入。
. 選擇選單 image:workloads-menu-icon.png["漢堡選單圖示用於導航至儲存、EDA、AI、資料庫、VMware 和管理等工作負載。"] 然後選擇 *Databases* 。
. 從資料庫選單中，選擇*Inventory*。
. 在清單中，選擇資料庫引擎類型：*Microsoft SQL Server* 或 *Oracle*。
. 導覽至要延後配置的 SQL Server 執行個體或 Oracle 資料庫，選擇操作選單，然後選擇 *Well-architected*。
. 在 Well-architected 分析頁面上，向下捲動到要延遲的配置，選擇操作選單，然後選擇 *Dismiss*。
. 在 Dismiss 設定對話方塊中，選擇 *Dismiss* 選項，然後選擇 *Dismiss* 確認關閉。


.結果
Workload Factory 停止組態分析。

您可以隨時重新啟動分析。此組態不再計入總優化分數。



== 重新啟動已關閉的組態分析

您可以隨時重新啟動已關閉的組態分析。您可以選擇一個或多個組態進行重新啟動。

.步驟
. 使用其中一項登link:https://docs.netapp.com/us-en/workload-setup-admin/console-experiences.html["主控台體驗"^]入。
. 選擇選單 image:workloads-menu-icon.png["漢堡選單圖示用於導航至儲存、EDA、AI、資料庫、VMware 和管理等工作負載。"] 然後選擇 *Databases* 。
. 從資料庫選單中，選擇*Inventory*。
. 在清單中，選擇資料庫引擎類型：*Microsoft SQL Server* 或 *Oracle*。
. 導覽至要延後配置的 SQL Server 執行個體或 Oracle 資料庫，選擇操作選單，然後選擇 *Well-architected*。
. 在 Well-architected 分析頁面上，選擇 *Dismissed configuration* 以僅顯示已解除的設定。
. 選擇*重新啟動*以重新啟動已延遲或已取消的配置的配置分析。


.結果
Workload Factory 會重新啟動組態分析，並從現在開始每天執行一次。
